# Зеркало перед доменом (staging)

Идея: **не трогаешь прод**, пока не проверил копию на тех же сервисах, но на **отдельных URL и отдельной БД**. Потом один раз вливаешь в `main` — и только тогда обновляется **crm.crmandromeda.ru** и **api.…**.

Подробности инфраструктуры: [STAGING.md](STAGING.md).

---

## Если в браузере `ERR_NAME_NOT_RESOLVED` / «не удаётся найти IP»

Публичный DNS для зеркала **ещё не настроен** (так бывает, если записи не создавали у регистратора домена).

В панели DNS зоны **`crmandromeda.ru`** нужны **две A-записи** на IP VPS **`163.5.29.68`**:

| Имя (host / subdomain) | Тип | Значение |
| --- | --- | --- |
| `dev.crm` | A | `163.5.29.68` |
| `dev-api.crm` | A | `163.5.29.68` |

В разных панелях поле называется по-разному: «поддомен», «Host», «Имя» — главное, чтобы итоговые имена были **`dev.crm.crmandromeda.ru`** и **`dev-api.crm.crmandromeda.ru`**.

После сохранения подожди **5–30 минут** (иногда до часа), обнови страницу или проверь в терминале: `dig +short dev.crm.crmandromeda.ru @8.8.8.8` — должно вывести `163.5.29.68`.

Когда DNS уже отвечает, на сервере должен быть выполнен workflow **Setup staging infrastructure** (Caddy, БД `food_crm_staging`, env) — см. [STAGING.md](STAGING.md).

## Адреса зеркала

| Что | URL |
| --- | --- |
| CRM (как на проде, но черновик) | **https://dev.crm.crmandromeda.ru** |
| API для этого CRM | **https://dev-api.crm.crmandromeda.ru** |

База зеркала: **`food_crm_staging`** (не продовая `food_crm`).

## Как работать каждый день

1. **Ветка** — черновые правки в **`dev`** (или фича-ветка → PR в `dev`). Прод крутится с **`main`**.

2. **Бэкенд** — репозиторий `food-crm-backend`: push в **`dev`** → в GitHub Actions запусти workflow **Deploy backend staging** (или настрой автозапуск на push в `dev`, если уже включён).

3. **Фронт** — репозиторий `food-crm`: push в **`dev`** → **Deploy frontend staging**.

4. **Проверка** — открываешь **https://dev.crm.crmandromeda.ru**, логинишься **пользователем из staging-БД** (если админа ещё нет — на VPS в каталоге **staging-бэкенда** тот же `npm run admin:create` с `DATABASE_URL` на `food_crm_staging`, либо отдельная учётка по вашему процессу).

5. **На домен** — когда зеркало ок: **merge `dev` → `main`**, затем push **`main`** и обычный прод-деплой (сначала бэкенд, потом фронт), как в [DEPLOYMENT.md](DEPLOYMENT.md).

---

## Что не путать

- Правки только на Mac **без push** на зеркало **никого не видят** — зеркало живёт на сервере после деплоя ветки **`dev`**.
- Логин и пароль **на зеркале** и **на проде** — разные базы; создавай админа **для staging**, если нужен свой вход на `dev.crm…`.
