# Зеркало перед доменом (staging)

Идея: **не трогаешь прод**, пока не проверил копию на тех же сервисах, но на **отдельных внутренних портах и отдельной БД**. Потом один раз вливаешь в `main` — и только тогда обновляется **crm.crmandromeda.ru** и **api.…**.

Подробности инфраструктуры: [STAGING.md](STAGING.md).
Полная модель веток и checklist перед `main`: [RELEASE_WORKFLOW.md](RELEASE_WORKFLOW.md).

---

## Если нужно проверить зеркало

Публичных dev-доменов больше нет. Staging живёт только внутри VPS:

```bash
ssh deploy@163.5.29.68
curl -I http://127.0.0.1:3100
curl -fsS http://127.0.0.1:4100/api/v1/health
```

На сервере должен быть выполнен workflow **Setup staging infrastructure** (БД `food_crm_staging`, env, PM2) — см. [STAGING.md](STAGING.md).

---

## Адреса зеркала

| Что | URL |
| --- | --- |
| CRM (как на проде, но черновик) | **http://127.0.0.1:3100** на VPS |
| API для этого CRM | **http://127.0.0.1:4100** на VPS |

База зеркала: **`food_crm_staging`** (не продовая `food_crm`).

---

## Как работать каждый день

1. **Ветка** — фича-ветка → PR в **`dev`**. Прод крутится с **`main`**. Модель: `feature branch -> dev -> staging -> main -> production`.

2. **Бэкенд** — репозиторий `food-crm-backend`: push в **`dev`** → в GitHub Actions запусти workflow **Deploy backend staging** (или настрой автозапуск на push в `dev`, если уже включён).

3. **Фронт** — репозиторий `food-crm`: push в **`dev`** → **Deploy frontend staging**.

4. **Проверка** — на VPS проверяешь **http://127.0.0.1:3100** и логинишься **пользователем из staging-БД** через SSH-туннель или серверную smoke-проверку (если админа ещё нет — на VPS в каталоге **staging-бэкенда** тот же `npm run admin:create` с `DATABASE_URL` на `food_crm_staging`, либо отдельная учётка по вашему процессу).

5. **На домен** — когда зеркало ок: **merge `dev` → `main`**, затем push **`main`** и обычный прод-деплой (сначала бэкенд, потом фронт), как в [DEPLOYMENT.md](DEPLOYMENT.md).

---

## Что не путать

- Правки только на Mac **без push** на зеркало **никого не видят** — зеркало живёт на сервере после деплоя ветки **`dev`**.
- Логин и пароль **на зеркале** и **на проде** — разные базы; создавай админа **для staging**, если нужен свой вход в staging-контур.
